# Migrating from v0.3 to v0.4

This guide covers all breaking changes introduced in `@getufy/flint-ui` v0.4.0 and `@getufy/flint-ui-react` v0.4.0, with before/after code examples and a migration checklist.

---

## 1. Namespaced Custom Events (Breaking Change)

All custom events have been renamed from generic names (e.g. `change`, `input`, `close`) to namespaced names following the `flint-{component}-{event}` pattern. This eliminates collisions with native DOM events and makes event origins unambiguous when multiple components are composed together.

### Updating Lit Templates

```ts
// v0.3
html`
  <flint-select @change=${this._onChange}></flint-select>
  <flint-input @input=${this._onInput} @change=${this._onChange}></flint-input>
  <flint-dialog @close=${this._onClose}></flint-dialog>
`

// v0.4
html`
  <flint-select @flint-select-change=${this._onChange}></flint-select>
  <flint-input @flint-input-input=${this._onInput} @flint-input-change=${this._onChange}></flint-input>
  <flint-dialog @flint-dialog-close=${this._onClose}></flint-dialog>
`
```

### Updating Vanilla JavaScript

```js
// v0.3
selectEl.addEventListener('change', handler);
dialogEl.addEventListener('close', handler);

// v0.4
selectEl.addEventListener('flint-select-change', handler);
dialogEl.addEventListener('flint-dialog-close', handler);
```

### Updating React Wrappers

React event handler props have been renamed to `onFlint{Component}{Event}`:

```tsx
// v0.3
<FlintSelect onChange={handleChange} />
<FlintTextField onChange={handleChange} onInput={handleInput} />
<FlintDialog onClose={handleClose}>...</FlintDialog>
<FlintSwitch onChange={handleChange} />

// v0.4
<FlintSelect onFlintSelectChange={handleChange} />
<FlintTextField onFlintTextFieldChange={handleChange} onFlintTextFieldInput={handleInput} />
<FlintDialog onFlintDialogClose={handleClose}>...</FlintDialog>
<FlintSwitch onFlintSwitchChange={handleChange} />
```

The React package also exports typed event constant objects for programmatic use:

```ts
import { FlintSelectEvents } from '@getufy/flint-ui-react/events';

// FlintSelectEvents.CHANGE === 'flint-select-change'
ref.current.addEventListener(FlintSelectEvents.CHANGE, handler);
```

### Finding Events to Rename

**Step 1: Lit / vanilla JS event listeners**

```bash
# Find old-style event listeners
grep -rn "addEventListener\(.\(change\|input\|close\|open\|click\)'" src/

# Find old-style Lit event bindings on flint elements
grep -rn '@change=\|@input=\|@close=\|@open=' src/ --include='*.ts'
```

**Step 2: React event handler props**


After updating the packages, run `tsc --noEmit`. The React wrapper types will flag any remaining uses of the old prop names as compile errors.

## 2. Two-File Split Architecture (Breaking Change for Deep Imports)

Components have been split from a single file into two files:

- **`flint-<name>.component.ts`** -- Pure class definition extending `FlintElement`. No custom element registration.
- **`flint-<name>.ts`** -- Imports the class, calls `.define()` to register the custom element, and re-exports the class.

```
# v0.3 structure
src/button/flint-button.ts          # class + @customElement('flint-button')

# v0.4 structure
src/button/flint-button.component.ts  # export class FlintButton extends FlintElement { ... }
src/button/flint-button.ts            # imports FlintButton, calls FlintButton.define('flint-button'), re-exports
```

### Impact on Barrel Imports (No Change Needed)

If you import from the package entry point or subpath exports, nothing changes:

```ts
// These still work exactly the same
import { FlintButton } from '@getufy/flint-ui';
import { FlintButton } from '@getufy/flint-ui/button/flint-button';
```

### Impact on Deep Internal Imports

If you were importing internal files by their full dist path (bypassing the `exports` map), the registration file is still at the same location and re-exports the class:

```ts
// v0.3
import { FlintButton } from '@getufy/flint-ui/dist/button/flint-button.js';

// v0.4 — still works (the .ts file re-exports from .component.ts)
import { FlintButton } from '@getufy/flint-ui/dist/button/flint-button.js';

// New in v0.4: import just the class without triggering registration
import { FlintButton } from '@getufy/flint-ui/dist/button/flint-button.component.js';
```

### Why This Matters

The two-file split enables:
- **Tree-shaking**: Bundlers can import the class without the `customElements.define()` side effect.
- **Autoloader compatibility**: The autoloader dynamically imports only the registration file when a tag appears in the DOM.
- **Manual registration**: Advanced users can import just the class and register it under a different tag name.

## 3. FlintElement Base Class (Breaking Change for Subclassing)

All components now extend `FlintElement` instead of `LitElement` directly. `FlintElement` is a thin extension of `LitElement` that adds:

- **`static dependencies`** -- A `Record<string, typeof FlintElement>` map of tag names to component classes. Dependencies listed here are auto-registered in the constructor.
- **`static define(name, elementClass?, options?)`** -- A safe registration method that handles duplicate registrations gracefully, following the [Shoelace pattern](https://shoelace.style).

### If You Only Use Flint Components

**No action required.** `FlintElement` extends `LitElement`, so the public API is unchanged.

### If You Extend Flint Components

Your subclass still works because `FlintElement` is a transparent extension of `LitElement`:

```ts
// v0.3
import { FlintButton } from '@getufy/flint-ui';

class MyButton extends FlintButton {
  // FlintButton extended LitElement directly
}
customElements.define('my-button', MyButton);

// v0.4 — recommended pattern
import { FlintButton } from '@getufy/flint-ui/button/flint-button.component';

class MyButton extends FlintButton {
  // FlintButton now extends FlintElement (which extends LitElement)
  // Declare sub-component dependencies your template uses
  static dependencies = {
    'flint-icon': FlintIcon,
  };
}
// Use .define() instead of customElements.define() for safe registration
MyButton.define('my-button');
```

The `static define()` method is preferred over `customElements.define()` because it silently handles the case where the tag name is already registered (e.g. in test environments or micro-frontend setups where multiple bundles load the same component).

## 4. `sideEffects` Removal from package.json

In v0.3, `package.json` declared `"sideEffects": ["*.css"]`. This caused bundlers (Webpack, Rollup, Vite) to incorrectly tree-shake away component registration files, since they only have the side effect of calling `customElements.define()`. Components would silently fail to render in production builds.

In v0.4, the `sideEffects` field was **removed entirely**, which fixes the production build issue.

### What You Need to Do

**Remove workarounds.** If you were adding explicit side-effect imports or bundler config to prevent tree-shaking of component registrations, you can remove them:

```ts
// v0.3 workaround — no longer needed
import { FlintButton } from '@getufy/flint-ui';
import '@getufy/flint-ui/button/flint-button'; // forced side-effect import to prevent tree-shaking

// v0.4 — just the barrel import is sufficient
import { FlintButton } from '@getufy/flint-ui';
```

> **Note:** In v0.6.0, a granular `sideEffects` array was re-added that correctly marks registration files (`./dist/*/flint-*.js`), CSS files, and `suppress-warnings.js` as side effects. If you upgrade directly to v0.6.0+, you get proper tree-shaking without the v0.3 bug.

## 5. Other Breaking Changes

### CSS `overflow: hidden` to `overflow: clip`

The `overflow` property was changed from `hidden` to `clip` on **FlintCard**, **FlintDialog**, and **FlintMenu**.

`overflow: clip` does not create a scroll container, so `scrollIntoView()` calls inside these components will no longer be intercepted. If your layout relied on the `overflow: hidden` behavior to contain scrolling, override it:

```css
flint-card::part(base) {
  overflow: hidden;
}
```

### FlintTextField Default Margin

The default bottom margin changed from `16px` to `0`. If your layout relied on the spacing:

```css
flint-text-field {
  --flint-text-field-margin-bottom: 16px;
}
```

### FlintMenu Sizing

`FlintMenu` now defaults to `min-width: 200px` and `width: max-content`. If your menus were narrower:

```css
flint-menu {
  min-width: auto;
}
```

### FlintMenuItem Width

`width: 100%` changed to `min-width: 100%`, which allows items to grow beyond the menu container width. This is generally the desired behavior, but if you need items clamped:

```css
flint-menu-item {
  max-width: 100%;
}
```

## 6. New Features Worth Adopting

These are not breaking changes, but v0.4.0 introduces features worth knowing about.

### Autoloader

Instead of importing every component individually, use the autoloader for CDN or lazy-loading scenarios:

```html
<script type="module">
  import '@getufy/flint-ui/autoloader';
</script>

<!-- Components register themselves when they appear in the DOM -->
<flint-button variant="primary">Click me</flint-button>
```

The autoloader uses a `MutationObserver` to detect `flint-*` elements and dynamically imports their registration modules.

### `hoist` Property

Dropdowns and popups on `flint-select`, `flint-tooltip`, `flint-hover-card`, and `flint-date-picker` now support `hoist` to escape `overflow: hidden`/`clip` containers using `position: fixed`:

```html
<flint-card>
  <!-- Dropdown escapes the card's overflow: clip -->
  <flint-select hoist></flint-select>
</flint-card>
```

### `::part()` CSS Selectors

20+ components now expose shadow DOM parts for external styling:

```css
flint-button::part(base)    { border-radius: 999px; }
flint-input::part(input)    { font-family: monospace; }
flint-dialog::part(panel)   { max-width: 800px; }
flint-select::part(trigger) { border-color: var(--flint-primary-color); }
```

### Localization via `LocalizeController`

Components with UI strings now use `LocalizeController`. Override strings per-locale:

```ts
import { registerTranslation } from '@getufy/flint-ui';

registerTranslation({
  $code: 'es',
  $name: 'Spanish',
  // component-specific keys...
});
```

### Animation Registry

Customize open/close/show/hide animations globally or per-instance:

```ts
import { setDefaultAnimation } from '@getufy/flint-ui';

setDefaultAnimation('dialog.show', {
  keyframes: [
    { opacity: 0, transform: 'scale(0.9)' },
    { opacity: 1, transform: 'scale(1)' },
  ],
  options: { duration: 200, easing: 'ease-out' },
});
```

### Form-Associated Custom Elements

Form components now participate in native `<form>` submission via `ElementInternals`:

```html
<form>
  <flint-input name="email" required></flint-input>
  <flint-select name="role" required></flint-select>
  <flint-button type="submit">Submit</flint-button>
</form>
```

### Preset Color Palettes

Six theme presets are available as single CSS imports:

```ts
import '@getufy/flint-ui/theme-violet.css';
// Also: theme-teal.css, theme-rose.css, theme-amber.css, theme-emerald.css, theme-slate.css
```

Apply via class or data attribute on any container:

```html
<div class="flint-theme-violet">
  <!-- All flint components inside use the violet palette -->
</div>
```

### Custom Elements Manifest

v0.4.0 ships `dist/custom-elements.json` (CEM) for IDE autocompletion:

- **VS Code**: Use the `vscode.html-custom-data.json` file
- **JetBrains**: Use the `web-types.json` file

## Migration Checklist

- [ ] Update `@getufy/flint-ui` to `^0.4.0` in `package.json`
- [ ] Update `@getufy/flint-ui-react` to `^0.4.0` if using React wrappers
- [ ] Rename all custom event listeners to `flint-{component}-{event}` format (see tables above)
- [ ] Rename React event handler props to `onFlint{Component}{Event}` format
- [ ] Run `tsc --noEmit` to catch remaining type errors from old event prop names
- [ ] Check layouts that relied on FlintTextField bottom margin (now `0` by default)
- [ ] Check layouts with menus narrower than 200px
- [ ] Remove any workarounds for the v0.3 `sideEffects` tree-shaking bug
- [ ] If extending Flint components, switch from `@customElement()` to `.define()`
- [ ] Test production build to verify components render correctly
- [ ] (Optional) Adopt `hoist`, `::part()`, localization, animation registry, or autoloader
