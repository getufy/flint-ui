# Migrating from v0.3 to v0.4

This guide covers the breaking changes and new features introduced in `@getufy/flint-ui` v0.4.0 and `@getufy/flint-ui-react` v0.4.0.

## 1. Event Naming Convention (Breaking Change)

All custom events now follow the `flint-{component}-{event}` pattern at the DOM level. This eliminates conflicts with native DOM events and ensures consistent behavior across frameworks.

### What changed

**DOM events (Lit / vanilla JS):**

| Component | Before (v0.3) | After (v0.4) |
|-----------|---------------|--------------|
| `flint-text-field` | `change`, `input` | `flint-text-field-change`, `flint-text-field-input` |
| `flint-date-picker` | `change` | `flint-date-picker-change` |
| `flint-time-picker` | `change` | `flint-time-picker-change` |
| `flint-transfer-list` | `change` | `flint-transfer-list-change` |
| `flint-chip` | `click` | `flint-chip-click`, `flint-chip-delete` |
| `flint-dialog` | `close` | `flint-dialog-close` |
| `flint-drawer` | `close` | `flint-drawer-close` |
| `flint-select` | `change` | `flint-select-change` |
| `flint-speed-dial` | `open`, `close` | `flint-speed-dial-open`, `flint-speed-dial-close` |
| `flint-hover-card` | `open`, `close` | `flint-hover-card-open`, `flint-hover-card-close` |
| `flint-menu` | `close` | `flint-menu-close` |
| `flint-snackbar` | `close` | `flint-snackbar-close` |
| `flint-textarea` | `input` | `flint-textarea-input` |

### React wrapper event handler renames

If you use `@getufy/flint-ui-react`, all event handler props have been renamed from React-conflicting names to the `onFlint{Component}{Event}` convention:

| Component | Before (v0.3) | After (v0.4) |
|-----------|---------------|--------------|
| `FlintTextField` | `onInput`, `onChange` | `onFlintTextFieldInput`, `onFlintTextFieldChange` |
| `FlintDatePicker` | `onChange` | `onFlintDatePickerChange` |
| `FlintTimePicker` | `onChange` | `onFlintTimePickerChange` |
| `FlintTransferList` | `onChange` | `onFlintTransferListChange` |
| `FlintChip` | `onClick` | `onFlintChipClick`, `onFlintChipDelete` |
| `FlintDialog` | `onClose` | `onFlintDialogClose` |
| `FlintDrawer` | `onClose` | `onFlintDrawerClose` |
| `FlintSelect` | `onChange` | `onFlintSelectChange` |
| `FlintSpeedDial` | -- | `onFlintSpeedDialOpen`, `onFlintSpeedDialClose` |
| `FlintMobileStepper` | -- | `onFlintMobileStepBack`, `onFlintMobileStepNext` |

### Migration steps

**Step 1: Find old event listeners in vanilla JS / Lit code**

Search for `addEventListener` calls using the old bare event names:

```bash
# Find old-style event listeners
grep -rn "addEventListener\(.\(change\|input\|close\|open\|click\)'" src/
```

Replace them with the namespaced equivalents:

```js
// Before (v0.3)
textField.addEventListener('change', handleChange);
dialog.addEventListener('close', handleClose);

// After (v0.4)
textField.addEventListener('flint-text-field-change', handleChange);
dialog.addEventListener('flint-dialog-close', handleClose);
```

**Step 2: Find old event handlers in Lit templates**

```bash
# Find old-style Lit event bindings
grep -rn '@change=\|@input=\|@close=\|@open=\|@click=' src/ --include='*.ts'
```

Update the bindings:

```ts
// Before (v0.3)
html`<flint-text-field @change=${this._onChange}></flint-text-field>`
html`<flint-dialog @close=${this._onClose}></flint-dialog>`

// After (v0.4)
html`<flint-text-field @flint-text-field-change=${this._onChange}></flint-text-field>`
html`<flint-dialog @flint-dialog-close=${this._onClose}></flint-dialog>`
```

**Step 3: Find old React event handler props**

```bash
# Find old-style React handler props on Flint components
grep -rn 'onChange=\|onInput=\|onClose=\|onClick=' src/ --include='*.tsx'
```

Update the props:

```tsx
// Before (v0.3)
<FlintTextField onChange={handleChange} onInput={handleInput} />
<FlintDialog onClose={handleClose}>...</FlintDialog>

// After (v0.4)
<FlintTextField onFlintTextFieldChange={handleChange} onFlintTextFieldInput={handleInput} />
<FlintDialog onFlintDialogClose={handleClose}>...</FlintDialog>
```

**Step 4: Let TypeScript catch the rest**

After updating the packages, run `tsc --noEmit`. The React wrapper types will flag any remaining uses of the old prop names as compile errors, so TypeScript serves as a safety net for anything you missed.

## 2. `sideEffects` Configuration Change

In v0.3, the core package declared `"sideEffects": ["*.css"]` in `package.json`. In v0.4, the `sideEffects` field has been **removed entirely**.

### Why this changed

Web component registration via `customElements.define()` is inherently a side effect. The previous `sideEffects` configuration could cause bundlers (Webpack, Rollup, Vite) to incorrectly tree-shake away component registrations in production builds, leading to undefined custom elements at runtime.

### What you need to do

If you were working around tree-shaking issues (e.g., adding explicit imports to prevent components from being dropped), you can remove those workarounds. Subpath imports now work correctly in production builds without special configuration:

```js
// Per-component imports still work and are recommended for smaller bundles
import '@getufy/flint-ui/button/flint-button';
import '@getufy/flint-ui/dialog/flint-dialog';
```

No changes to your bundler configuration should be necessary.

## 3. Other Breaking Changes

### CSS overflow change

The `overflow` property was changed from `hidden` to `clip` on several components: **Card**, **Dialog**, **Menu**, **ResizableGroup**, **ResizablePanel**, and **SplitPanel**.

This is a subtle behavioral difference: `overflow: clip` does not create a scroll container, so `scrollIntoView()` calls inside these components will no longer be intercepted. If you relied on the `overflow: hidden` behavior to contain scrolling, add `overflow: hidden` to your component via CSS custom properties or a wrapper element.

### TextField default margin

`FlintTextField` default bottom margin changed from `0 0 16px` to `var(--flint-text-field-margin-bottom, 0)`. If your layout relied on the old 16px bottom margin, set the CSS custom property:

```css
flint-text-field {
  --flint-text-field-margin-bottom: 16px;
}
```

### Menu sizing

`FlintMenu` now has `min-width: 200px` and `width: max-content` by default. A new `--flint-menu-max-width` CSS custom property is available to control the maximum width. If your menus were narrower than 200px, you may need to adjust:

```css
flint-menu {
  --flint-menu-max-width: 150px;
  min-width: auto;
}
```

## 4. New Features

### Autoloader for CDN and lazy-loading

A new autoloader entry point automatically imports and registers `flint-*` components as they appear in the DOM. This is useful for CDN usage, server-rendered HTML, or lazy-loading scenarios where you don't want to manually import each component:

```html
<script type="module">
  import '@getufy/flint-ui/autoloader';
</script>

<!-- Components are automatically registered when the DOM parser encounters them -->
<flint-button variant="contained">Click me</flint-button>
<flint-dialog>
  <flint-card>Hello</flint-card>
</flint-dialog>
```

The autoloader uses a `MutationObserver` to watch for new elements and dynamically imports the corresponding module. It skips elements that are already defined and deduplicates concurrent imports.

### Typed `CustomEvent<T>` detail interfaces

All events now ship with typed detail interfaces. In TypeScript, you get full IntelliSense on event payloads:

```ts
// DOM usage
textField.addEventListener('flint-text-field-change', (e) => {
  // e.detail is fully typed
  console.log(e.detail.value);
});

// React usage
<FlintDialog
  onFlintDialogClose={(e) => {
    // e.detail.open is typed as boolean
    if (!e.detail.open) {
      console.log('Dialog closed');
    }
  }}
/>
```

### `FlintElement` base class with static dependencies

A new `FlintElement` base class (extending `LitElement`) provides a `static dependencies` pattern for declaring sub-component dependencies. This ensures child components are always registered when a parent component is used, without relying on import side effects:

```ts
import { FlintElement } from '@getufy/flint-ui';
import { FlintButton } from '@getufy/flint-ui/button/flint-button';
import { FlintIcon } from '@getufy/flint-ui/icon/flint-icon';

class MyComponent extends FlintElement {
  static dependencies = {
    'flint-button': FlintButton,
    'flint-icon': FlintIcon,
  };
}
```

When `MyComponent` is instantiated, it automatically calls `customElements.define()` for each dependency that is not already registered. This is particularly useful with the autoloader or in micro-frontend architectures where component registration order is not guaranteed.

## Upgrade checklist

- [ ] Update `@getufy/flint-ui` and `@getufy/flint-ui-react` to `^0.4.0`
- [ ] Rename all custom event listeners to use `flint-{component}-{event}` pattern
- [ ] Rename React event handler props to `onFlint{Component}{Event}` pattern
- [ ] Run `tsc --noEmit` to catch any remaining type errors
- [ ] Check layouts that depend on TextField bottom margin (now `0` by default)
- [ ] Check layouts with menus narrower than 200px
- [ ] Remove any `sideEffects` workarounds in your bundler config
- [ ] Test production build to verify components render correctly
