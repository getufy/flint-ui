# Migrating from v0.4.0 to v0.5.0

This guide covers breaking changes in `@getufy/flint-ui` v0.5.0 and how to update your code.

---

## 1. FlintAccordion: renamed sub-components

The accordion sub-component tags have been renamed to follow a `summary`/`details` pattern, replacing the previous `item`/`trigger`/`content` naming.

| v0.4.0 tag | v0.5.0 tag |
|---|---|
| `<flint-accordion-item>` | Removed — use `<flint-accordion>` directly |
| `<flint-accordion-trigger>` | `<flint-accordion-summary>` |
| `<flint-accordion-content>` | `<flint-accordion-details>` |

The `<flint-accordion-actions>` sub-component is new in v0.5.0 for grouping action buttons inside an accordion.

### Before (v0.4.0)

```html
<flint-accordion>
  <flint-accordion-item>
    <flint-accordion-trigger>Section Title</flint-accordion-trigger>
    <flint-accordion-content>
      <p>Accordion content goes here.</p>
    </flint-accordion-content>
  </flint-accordion-item>
</flint-accordion>
```

### After (v0.5.0)

```html
<flint-accordion>
  <flint-accordion-summary>Section Title</flint-accordion-summary>
  <flint-accordion-details>
    <p>Accordion content goes here.</p>
  </flint-accordion-details>
</flint-accordion>
```

### Class name changes

| v0.4.0 class | v0.5.0 class |
|---|---|
| `FlintAccordionItem` | Removed |
| `FlintAccordionTrigger` | `FlintAccordionSummary` |
| `FlintAccordionContent` | `FlintAccordionDetails` |

Update any JavaScript/TypeScript imports accordingly:

```diff
- import { FlintAccordionTrigger, FlintAccordionContent } from '@getufy/flint-ui';
+ import { FlintAccordionSummary, FlintAccordionDetails } from '@getufy/flint-ui';
```

### React wrapper changes

```diff
- import { FlintAccordionTrigger, FlintAccordionContent } from '@getufy/flint-ui-react';
+ import { FlintAccordionSummary, FlintAccordionDetails } from '@getufy/flint-ui-react';
```

### Search and replace

Run these across your project to catch all usages:

| Find | Replace |
|---|---|
| `flint-accordion-item` | `flint-accordion` |
| `flint-accordion-trigger` | `flint-accordion-summary` |
| `flint-accordion-content` | `flint-accordion-details` |
| `FlintAccordionItem` | (remove wrapper, use `FlintAccordion` directly) |
| `FlintAccordionTrigger` | `FlintAccordionSummary` |
| `FlintAccordionContent` | `FlintAccordionDetails` |

## 2. FlintTabs: `<flint-tab-list>` wrapper required

Tab elements must now be wrapped in a `<flint-tab-list>` component. This new wrapper handles keyboard navigation, scroll buttons, and the active-tab indicator. Tabs placed directly inside `<flint-tabs>` without the list wrapper will not render correctly.

### Before (v0.4.0)

```html
<flint-tabs value="tab1">
  <flint-tab value="tab1" label="First"></flint-tab>
  <flint-tab value="tab2" label="Second"></flint-tab>
  <flint-tab-panel value="tab1">Content 1</flint-tab-panel>
  <flint-tab-panel value="tab2">Content 2</flint-tab-panel>
</flint-tabs>
```

### After (v0.5.0)

```html
<flint-tabs value="tab1">
  <flint-tab-list>
    <flint-tab value="tab1">First</flint-tab>
    <flint-tab value="tab2">Second</flint-tab>
  </flint-tab-list>
  <flint-tab-panel value="tab1">Content 1</flint-tab-panel>
  <flint-tab-panel value="tab2">Content 2</flint-tab-panel>
</flint-tabs>
```

Note two changes in this example:
1. `<flint-tab>` elements are wrapped in `<flint-tab-list>`.
2. The `label` attribute has been replaced with slotted content (see next section).

### React

```diff
  <FlintTabs value="tab1">
+   <FlintTabList>
-     <FlintTab value="tab1" label="First" />
-     <FlintTab value="tab2" label="Second" />
+     <FlintTab value="tab1">First</FlintTab>
+     <FlintTab value="tab2">Second</FlintTab>
+   </FlintTabList>
    <FlintTabPanel value="tab1">Content 1</FlintTabPanel>
    <FlintTabPanel value="tab2">Content 2</FlintTabPanel>
  </FlintTabs>
```

### FlintTabList props

Properties that previously lived on `<flint-tabs>` for controlling tab layout are now also mirrored on `<flint-tab-list>`, and `<flint-tabs>` forwards them automatically. You do not need to set them in both places. The following props are synced from `<flint-tabs>` to `<flint-tab-list>`:

- `orientation` (`"horizontal"` | `"vertical"`)
- `variant` (`"standard"` | `"fullWidth"` | `"scrollable"`)
- `centered`
- `scroll-buttons` (`"auto"` | `"false"`)

## 3. FlintTab: `label` attribute replaced with slotted content

The `label` attribute on `<flint-tab>` has been removed. Use the default slot (child content) instead. This enables richer tab labels including icons and custom markup.

### Before (v0.4.0)

```html
<flint-tab value="settings" label="Settings"></flint-tab>
```

### After (v0.5.0)

```html
<flint-tab value="settings">Settings</flint-tab>
```

### With an icon

```html
<flint-tab value="settings">
  <svg slot="icon" ...></svg>
  Settings
</flint-tab>
```

## 4. FlintDialog: `aria-labelledby` replaced with dynamic `aria-label`

The dialog panel previously used `aria-labelledby` referencing the title element's ID across the shadow DOM boundary. This caused an `aria-dialog-name` axe-core a11y violation because `aria-labelledby` cannot reference elements inside a different shadow root.

In v0.5.0, the dialog reads the text content of the slotted `<flint-dialog-title>` via a `slotchange` handler and sets a dynamic `aria-label` attribute on the dialog panel instead.

### What you need to do

**Nothing in most cases.** This change is internal to the component. However, if you were relying on the `aria-labelledby` attribute for custom styling or testing selectors, update those references:

```diff
- // v0.4.0: dialog panel had aria-labelledby="some-id"
- dialog.shadowRoot.querySelector('[aria-labelledby]')

+ // v0.5.0: dialog panel has aria-label="Dialog Title Text"
+ dialog.shadowRoot.querySelector('[aria-label]')
```

If your dialog does not contain a `<flint-dialog-title>`, the `aria-label` will be empty. To ensure accessibility, always include a title:

```html
<flint-dialog open>
  <flint-dialog-title>Confirm Action</flint-dialog-title>
  <flint-dialog-content>
    <flint-dialog-content-text>Are you sure?</flint-dialog-content-text>
  </flint-dialog-content>
  <flint-dialog-actions>
    <flint-button variant="text">Cancel</flint-button>
    <flint-button variant="primary">Confirm</flint-button>
  </flint-dialog-actions>
</flint-dialog>
```

## 5. FlintSelect: `hoist` default changed to `true`

The `hoist` property on `<flint-select>` now defaults to `true`. When hoisted, the dropdown uses `position: fixed` so it can escape containers with `overflow: hidden` or `overflow: clip` (such as dialogs and cards).

### What you need to do

**Nothing in most cases.** The new default matches how most applications use `<flint-select>` (inside cards, dialogs, or scrollable containers).

If you were relying on the dropdown being positioned absolutely within its parent container, explicitly set `hoist` to `false`:

```diff
- <flint-select .options=${options}></flint-select>
+ <flint-select .options=${options} ?hoist=${false}></flint-select>
```

Or in React:

```diff
- <FlintSelect options={options} />
+ <FlintSelect options={options} hoist={false} />
```

## 6. Eight components migrated to `FlintElement.define()`

The following components have been migrated from the `@customElement` decorator to `FlintElement.define()` for registration:

- FlintButton
- FlintCard
- FlintDatePicker
- FlintHoverCard
- FlintInput
- FlintSwitch
- FlintTabs (and FlintTab, FlintTabList, FlintTabPanel)
- FlintTooltip

### What you need to do

**Nothing if you import from the barrel export** (`@getufy/flint-ui` or `@getufy/flint-ui-react`). The registration files handle calling `.define()` automatically.

If you were importing the component class directly from a `.component.ts` file (e.g., for subclassing or manual registration), be aware that these classes no longer carry the `@customElement('flint-...')` decorator. You must either:

1. Import from the registration file (recommended), which calls `.define()` as a side effect:

   ```ts
   // This import registers the element automatically
   import '@getufy/flint-ui/dist/button/flint-button.js';
   ```

2. Or register manually after importing the bare class:

   ```ts
   import { FlintButton } from '@getufy/flint-ui/dist/button/flint-button.component.js';
   FlintButton.define('flint-button');
   ```

### Benefits of this migration

- **Deduplication safety**: if the same component is imported from multiple paths, `FlintElement.define()` silently skips re-registration instead of throwing.
- **Dependency auto-registration**: components that use other `flint-*` elements in their templates declare them in `static dependencies`, and those are registered automatically in the constructor.
- **Tree-shaking friendly**: bare class imports do not trigger side effects.

## Migration checklist

- [ ] Rename `flint-accordion-item`/`trigger`/`content` tags and classes to the new `summary`/`details` pattern
- [ ] Wrap `<flint-tab>` elements in `<flint-tab-list>`
- [ ] Replace `label` attribute on `<flint-tab>` with slotted text content
- [ ] Verify dialog accessibility (ensure `<flint-dialog-title>` is present)
- [ ] Check `<flint-select>` usage if you need non-hoisted dropdowns -- set `hoist={false}` explicitly
- [ ] If importing `.component.ts` files directly, ensure you call `.define()` or import the registration file
